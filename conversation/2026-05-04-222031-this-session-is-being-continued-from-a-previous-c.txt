● All changes applied cleanly. Summary:

  train.py (one switch + one VRAM accommodation)
  - MODEL_YAML: yolo11n.yaml → yolo11m.yaml (~7× more params, much better
  small-object generalization).
  - BATCH: 8 → 4 — yolo11m at imgsz=1280 needs ~7-8 GB just for activations;
  batch=4 is what actually fits an 8 GB RTX 4000. If you have 16 GB+ available,
  push it back to 8.

  generate_column.py (purely additive — original code path still triggers on the
   negation of the new probabilistic branches)
  - Added LIGHT_FILL_PROB = 0.25 + a new pale-gray fill range (190-220) in
  place_column. The default medium-gray (130-185) still fires 75% of the time —
  nothing was removed.
  - Added SMALL_COL_PROB = 0.20 + SMALL_COL_MIN/MAX_SIZE = 12-20 constants. At
  the column placement site, square/unshaded-square columns (cls 0 / cls 5) are
  shrunk into the small band 20% of the time. Default size band still fires the
  other 80%.
  - Did not touch rect (cls 1/6), round (cls 2/4), or combined (cls 3) sizing —
  those depend on a larger floor for detectability and shouldn't be shrunk.

  Couple of things to know before you regenerate + retrain:
  - yolo11m takes roughly 3-4× longer per epoch than yolo11n on the same VRAM,
  so plan for the run to take longer.
  - yolo11m.yaml is bundled in ultralytics (no download), so the first call will
   work offline like before.
  - If batch=4 still OOMs on your card, drop IMGSZ to 1024 or batch to 2; let me
   know and I'll wire that up.

  Order of operations: regenerate dataset (python3 generate_column.py), then
  retrain (python3 train.py). Want me to kick those off, or eyeball a sample
  tile first to confirm the light-fill / small-square variants look right?
