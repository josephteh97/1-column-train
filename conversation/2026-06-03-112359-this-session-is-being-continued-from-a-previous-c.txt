  What happened

  Why it happened
  Two compounding causes:
  1. AMP overflow. amp=True (the default) and the log shows AMP: checks passed

  single most common YOLO-NaN cause.

  gradients should be tiny by epoch 4, but with lr=0.01 a single big-gradient
  batch can lurch the weights into a regime where the next forward overflows.

  The combo: fast convergence + high LR + AMP fp16 ceiling = epoch-4 NaN. This

‚óŹ Found it. The output makes the cause unambiguous:

  raw detections                       : 1162
      (stair regions detected: 19)
  (0) after stair-mask                  : 0

  Filter 0 (STAIR-MASK) eats all 1162 detections. The stair detector is
  over-firing ‚ÄĒ it claims 19 stair regions on this plan, and between them they
  cover essentially the whole drawing, so every column centre lands "inside a
  stair" and gets dropped.

  Why it's over-firing

  _detect_stair_regions runs cv2.HoughLinesP on the full 13480√ó9536 image with
  threshold=30, minLineLength=15, then clusters any nearly-parallel
  horizontal/vertical lines spaced 12‚Äď35 px apart with STAIR_MIN_TREADS=3. On a
  real architectural plan that's an enormous catchment:

  - Wall hatch patterns, beam crossings, dimension tick chains, grid extension
  lines, text underlines, drawing-border decoration ‚ÄĒ all produce dense
  parallel-line clusters at 12‚Äď35 px pitch.
  - The cluster loop has no upper bound on the cluster's perpendicular span. A
  run of grid lines at ~30 px pitch across half the sheet becomes one giant
  "stair."
  - STAIR_REGION_PAD_PX = 10 then inflates each region. 19 of them √ó huge area =
  everything is masked.

  There's also a real bug in the cluster step: when a line's gap is < MIN, it
  does i += 1 but j is left pointing somewhere ahead ‚ÄĒ the indexing isn't quite
  right, which causes overlapping/runaway clusters.

  How to fix

  Three viable directions, increasing in effort:

  1. Disable stair-mask entirely for now (set stair_regions = []) and rely on
  filters 1-5. The shape filter alone kills empty seam-ghosts and most stair
  T-junctions; aspect + size kill stair-tread bboxes. You can validate the rest

     Approach ‚ÄĒ the HITL review notebook (the remaining piece)

     Design choice: paginated thumbnail grid + side-panel context

     After post-processing, a real plan typically has 200-1000 surviving
     column detections. The reviewer needs to scan them fast AND see local
     context (is this really a column, or a wall corner that looks like
     one?). A paginated thumbnail grid with a side panel that opens
     the full plan at the focused detection is the right level of detail:

     - Each thumbnail: ~96 √ó 96 px crop around the detection (12 px pad),
     drawn with the bbox outlined in red.
     - Page size: 5 √ó 4 = 20 thumbnails per page (fits comfortably in
     the notebook viewport).
     - Per-thumbnail state cycles via a Dropdown widget below it:
     KEEP (default) / DELETE (mark as FP).
     - Page nav: prev / next buttons + page index label.
     - Save button: writes all DELETE marks to data/corrections.db in one
     transaction. Per-click DB writes are spammy and not undoable.

     v1 scope (what this notebook covers)

     Success criterion from the user: "YOLO absorbs the corrections and
     evolves and improves." That requires closing the loop in BOTH error
     directions ‚ÄĒ fixing FPs alone leaves the model stuck on its current
     recall, and fixing FNs alone leaves the residual FPs. So v1 must
     capture both:

     - DELETE (mark surviving detection as FP) ‚ÄĒ kills the residual
     FPs the post-processing pipeline missed. Primary action, surfaced
     in the per-thumbnail dropdown in the review grid.
     - ADD missed column ‚ÄĒ feeds FNs back into the corrections DB so
     the next retrain learns the missed pattern. Implemented as a
     separate cell that takes a list of (cx, cy, size_px) tuples the
     reviewer reads off the plan (image-viewer hover readout). Each entry
     becomes a synthetic bbox + record_add call. Lightweight; no
     click-on-image UI in v1.
     - EDIT bbox shape ‚ÄĒ out of scope for v1. In practice the model
     emits well-fitting bboxes once a detection survives the shape
     filter; the dominant correction is DELETE. The logger module
     supports record_edit for future use, but the v1 UI does not
     surface it.

     End-to-end loop closure (the user's actual goal): review ‚Üí DB ‚Üí
     scripts/retrain_yolo.py ‚Üí column_detect_ft_{ts}.pt ‚Üí manual copy
     to column_detect.pt ‚Üí re-run inference ‚Üí next review cycle. Each
     turn of the loop should reduce the per-plan FP/FN count.
